01 Overview & Providers

Elastic publishes two complementary Terraform providers. Together they give you full Infrastructure-as-Code coverage of both the Elastic Cloud control plane and the Elastic Stack itself.

☁️
elastic/ec
Manages Elastic Cloud infrastructure — deployments, serverless projects, traffic filters, remote clusters. Talks to the Elastic Cloud API.
🔍
elastic/elasticstack
Configures the Elastic Stack itself — index templates, ILM policies, users, roles, ingest pipelines, dashboards. Works on-prem or on cloud.

This tutorial focuses on elastic/ec. The final section shows how to chain both providers together for end-to-end deployments.

ℹ️
Supported Environments

The elastic/ec provider supports Elasticsearch Service (ESS), Elastic Cloud Enterprise (ECE), Elasticsearch Service Private (ESSP), and Elastic Serverless projects.

03 Provider Setup & Authentication

Create a versions.tf (or provider.tf) file to declare the provider and pin its version.

HCL versions.tf 
terraform {
  required_version = ">= 1.2.7"

  required_providers {
    ec = {
      source  = "elastic/ec"
      version = "~> 0.11"  # pin to latest minor
    }
  }
}

# Provider reads EC_API_KEY from environment automatically
provider "ec" {}
⚠️
Never Hard-Code Credentials

Do not put API keys directly in .tf files. Use the EC_API_KEY environment variable or a secrets manager like Vault.

Provider Configuration Options

endpointStringhttps://api.elastic-cloud.comOverride for ECE or private installations.

passwordString (sensitive)—Password for ECE / ESSP.

ECE / On-Prem Configuration

HCL provider.tf (ECE) 
provider "ec" {
  endpoint = "https://ece.mycompany.internal:12443"
  apikey   = var.ec_api_key
  insecure = true  # only for self-signed certs in dev
}

After creating your configuration, run terraform init to download the provider:

BASH 
terraform init

04 Your First Deployment

The ec_deployment resource creates a full Elastic Stack deployment. The ec_stack data source lets you reference the latest available stack version dynamically.

HCL main.tf 
# Look up the latest available stack version for this region
data "ec_stack" "latest" {
  version_regex = "latest"
  region        = "us-east-1"
}

# Minimal deployment: Elasticsearch + Kibana only
resource "ec_deployment" "example" {
  name                   = "my-first-deployment"
  region                 = "us-east-1"
  version                = data.ec_stack.latest.version
  deployment_template_id = "aws-io-optimized-v2"

  elasticsearch = {
    hot = {
      autoscaling = {}  # use template defaults
    }
  }

  kibana = {}
}

Key Deployment Arguments

ArgumentRequiredDescription
region✅ YesCloud region. Examples: us-east-1, gcp-europe-west3, azure-eastus. ECE uses ece-region.
version✅ YesElastic Stack version string. Use data.ec_stack.latest.version for auto-latest.
deployment_template_id✅ YesTemplate ID defining hardware profile. See templates below.
nameNoHuman-readable deployment name.
aliasNoShortname affecting resource URLs. Set "" to disable.
tagsNoMap of string tags applied to the deployment.

05 Topology & Sizing

From provider v0.6.0 onward, the elasticsearch block uses attribute syntax (= {...} not block syntax) and each data tier is configured separately. Available tiers: hot, warm, cold, frozen, coordinating, ml, master.

HCL main.tf — hot-warm architecture 
resource "ec_deployment" "hot_warm" {
  name                   = "hot-warm-logging"
  region                 = "us-east-1"
  version                = data.ec_stack.latest.version
  deployment_template_id = "aws-storage-optimized-v3"

  elasticsearch = {
    # Hot tier: recent/active data
    hot = {
      size             = "8g"     # memory per node
      size_resource    = "memory"
      zone_count       = 2
      autoscaling      = {}
    }

    # Warm tier: older, less-queried data
    warm = {
      size          = "4g"
      size_resource = "memory"
      zone_count    = 1
      autoscaling   = {}
    }

    # Cold tier: archived, infrequent access
    cold = {
      size          = "2g"
      size_resource = "memory"
      zone_count    = 1
      autoscaling   = {}
    }
  }

  kibana = {
    topology = {
      size       = "2g"
      zone_count = 1
    }
  }

  integrations_server = {}  # Fleet/APM for Elastic Stack 8.x+
}

Tier Size Reference

Size ValueMemoryTypical Use
"1g"1 GBDev / testing
"2g"2 GBSmall production, Kibana
"4g"4 GBWarm/cold tiers, light workloads
"8g"8 GBStandard hot tier
"16g"16 GBHeavy search, analytics
"32g"32 GBLarge clusters, ML nodes
"64g"64 GBMemory-intensive / ML
ℹ️
Dedicated Masters

The provider automatically adds a dedicated master tier when the number of data nodes in your cluster exceeds the threshold requiring it, and removes it again when not needed. You can also configure master = { ... } explicitly.

06 Autoscaling

Autoscaling allows Elastic Cloud to dynamically adjust tier capacity based on disk usage and ML demand. Each tier that supports autoscaling requires an autoscaling = {} block. You can optionally set min/max bounds.

HCL main.tf — autoscaling with bounds 
resource "ec_deployment" "autoscaled" {
  name                   = "autoscaled-cluster"
  region                 = "us-east-1"
  version                = data.ec_stack.latest.version
  deployment_template_id = "aws-io-optimized-v2"

  elasticsearch = {
    hot = {
      autoscaling = {
        autoscale       = true
        min_size        = "8g"
        max_size        = "64g"
        min_zone_count  = 2
        max_zone_count  = 3
      }
    }

    ml = {
      autoscaling = {
        autoscale = true
        min_size  = "0g"   # start at zero when no ML jobs
        max_size  = "32g"
      }
    }
  }

  kibana = {}
}
⚠️
ML Tier Drift

When autoscaling is enabled, all autoscaleable topology elements must be explicitly defined in the elasticsearch block. Omitting the ml block will cause persistent plan diffs.

07 Serverless Projects

The provider supports three Serverless project types, added in recent releases. Serverless projects are fully managed — no topology or sizing to configure.

ec_elasticsearch_project Serverless
Serverless Elasticsearch project for search and analytics workloads.
ec_observability_project Serverless
Serverless Observability project for logs, metrics, and APM.
ec_security_project Serverless
Serverless Security project for SIEM and threat detection.
HCL serverless.tf 
# Serverless Elasticsearch project
resource "ec_elasticsearch_project" "search" {
  name       = "my-search-project"
  region_id  = "aws-us-east-1"
}

# Serverless Observability project
resource "ec_observability_project" "obs" {
  name      = "platform-observability"
  region_id = "aws-us-east-1"
}

# Serverless Security project
resource "ec_security_project" "siem" {
  name      = "security-operations"
  region_id = "aws-us-east-1"
}

# Output the serverless endpoints
output "search_endpoint" {
  value = ec_elasticsearch_project.search.endpoints.elasticsearch
}

08 Traffic Filters

Traffic filters restrict inbound access to your deployment to specific IP ranges or VPC endpoints. The provider manages the full set of rules for a deployment — do not mix traffic_filter on ec_deployment with ec_deployment_traffic_filter_association for the same deployment.

HCL traffic.tf 
# Define an IP-based traffic filter
resource "ec_deployment_traffic_filter" "office" {
  name   = "office-ips"
  region = "us-east-1"
  type   = "ip"

  rule = [{
    source      = "203.0.113.0/24"
    description = "Sydney Office"
  }, {
    source      = "198.51.100.0/24"
    description = "Auckland Office"
  }]
}

# Attach the filter to your deployment
resource "ec_deployment" "secure" {
  name                   = "filtered-deployment"
  region                 = "us-east-1"
  version                = data.ec_stack.latest.version
  deployment_template_id = "aws-io-optimized-v2"

  traffic_filter = [
    ec_deployment_traffic_filter.office.id
  ]

  elasticsearch = { hot = { autoscaling = {} } }
  kibana        = {}
}

AWS PrivateLink Traffic Filter

HCL privatelink.tf 
resource "ec_deployment_traffic_filter" "privatelink" {
  name   = "aws-privatelink"
  region = "us-east-1"
  type   = "vpce"

  rule = [{
    source      = "vpce-0123456789abcdef0"
    description = "Production VPC Endpoint"
  }]
}

09 Remote Clusters (CCS/CCR)

Cross-Cluster Search and Cross-Cluster Replication require setting up remote cluster relationships. You can define these directly on the ec_deployment resource.

HCL remote_clusters.tf 
# Primary cluster
resource "ec_deployment" "primary" {
  name                   = "primary-cluster"
  region                 = "us-east-1"
  version                = data.ec_stack.latest.version
  deployment_template_id = "aws-io-optimized-v2"
  elasticsearch          = { hot = { autoscaling = {} } }
  kibana                 = {}
}

# Secondary cluster with remote reference to primary
resource "ec_deployment" "secondary" {
  name                   = "secondary-cluster"
  region                 = "us-east-1"
  version                = data.ec_stack.latest.version
  deployment_template_id = "aws-io-optimized-v2"

  elasticsearch = {
    hot = { autoscaling = {} }

    remote_cluster = [{
      deployment_id = ec_deployment.primary.id
      alias         = ec_deployment.primary.name
      ref_id        = ec_deployment.primary.elasticsearch.ref_id
    }]
  }

  kibana = {}
}

10 Combining with Elastic Stack Provider

The elastic/elasticstack provider configures resources inside an Elasticsearch cluster — index templates, ILM policies, users, ingest pipelines, and more. Chain it with elastic/ec outputs for full end-to-end IaC.

HCL versions.tf (both providers) 
terraform {
  required_version = ">= 1.2.7"
  required_providers {
    ec = {
      source  = "elastic/ec"
      version = "~> 0.11"
    }
    elasticstack = {
      source  = "elastic/elasticstack"
      version = "~> 0.11"
    }
  }
}

provider "ec" {}

# Wire the Stack provider using deployment outputs
provider "elasticstack" {
  elasticsearch {
    username  = ec_deployment.main.elasticsearch_username
    password  = ec_deployment.main.elasticsearch_password
    endpoints = [ec_deployment.main.elasticsearch.https_endpoint]
  }
}
HCL stack_config.tf — ILM + index template 
# Create an ILM policy via the Stack provider
resource "elasticstack_elasticsearch_index_lifecycle" "logs_policy" {
  name = "logs-30-day"

  hot = {
    min_age = "1d"
    rollover = {
      max_primary_shard_size = "50gb"
      max_age                = "7d"
    }
  }

  warm = {
    min_age    = "7d"
    shrink     = { number_of_shards = 1 }
    forcemerge = { max_num_segments = 1 }
  }

  delete = {
    min_age = "30d"
    delete  = {}
  }
}

# Create a component template
resource "elasticstack_elasticsearch_component_template" "logs_settings" {
  name = "logs-settings"

  template = {
    settings = jsonencode({
      index = {
        lifecycle = { name = "logs-30-day" }
        number_of_shards = "3"
      }
    })
  }
}

11 State Management & Import

If you have existing Elastic Cloud deployments not yet managed by Terraform, you can import them into state using the deployment ID.

BASH Import existing deployment 
# Import by deployment ID (found in Elastic Cloud console URL)
terraform import 'ec_deployment.my_deployment' abc123def456

# After import, run plan to confirm state is clean
terraform plan
⚠️
Import Limitations

State import does not capture user passwords or secret tokens. After import, terraform plan may show changes for empty/zero-size attributes (e.g. cold tier). These can safely be applied — only state is updated, the deployment itself does not change.

Remote State (Recommended for Teams)

HCL backend.tf — S3 remote state 
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "elastic/prod/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"
  }
}

Upgrading Versions

HCL main.tf — targeted version upgrade 
# Pin to a specific version for controlled upgrades
resource "ec_deployment" "prod" {
  version = "8.17.0"  # bump this to trigger upgrade
  # ...rest of config
}

Running terraform plan after updating the version string will show a detected change. terraform apply will perform a rolling upgrade.

A Appendix: Discover Templates & Instance Configs via API

Template IDs and instance configuration IDs change between Elastic Cloud versions and cloud providers. Rather than guessing from docs, always query the live API for the exact values valid in your region. The commands below work for any hyperscaler (AWS, Azure, GCP) and any region.

🚨
Why This Matters

The Elastic Cloud API marks older templates with "legacy": "true" in their metadata. Terraform will accept these values at plan time but fail at apply time with deployments.elasticsearch_using_legacy_dt errors. Always filter for non-legacy templates before writing your config.

Step 1 — List All Valid (Non-Legacy) Templates for a Region

Replace REGION with your target region code (e.g. azure-australiaeast, us-east-1, gcp-europe-west3).

BASH Step 1 — list non-legacy templates 
export REGION="azure-australiaeast"   # ← change this

curl -s -H "Authorization: ApiKey $EC_API_KEY" \
  "https://api.elastic-cloud.com/api/v1/deployments/templates?region=$REGION" \
  | jq '[.[] | select(.metadata != null)
    | select((.metadata
        | map(select(.key == "legacy" and .value == "true")))
        | length == 0)
    | {id: .id, name: .name}]'

Example output for azure-australiaeast:

JSON sample output 
[
  { "id": "azure-storage-optimized",      "name": "Storage optimized" },
  { "id": "azure-cpu-optimized",          "name": "CPU optimized" },
  { "id": "azure-vector-search-optimized", "name": "Vector Search optimized" },
  { "id": "azure-general-purpose",        "name": "General purpose" }
]

Step 2 — Inspect a Specific Template's Instance Configs

Once you've chosen a template ID from Step 1, pull its full definition to get the exact instance_configuration_id for each tier.

BASH Step 2 — get instance config IDs per tier 
export REGION="azure-australiaeast"
export TEMPLATE_ID="azure-general-purpose"  # ← from Step 1

curl -s -H "Authorization: ApiKey $EC_API_KEY" \
  "https://api.elastic-cloud.com/api/v1/deployments/templates?region=$REGION" \
  | jq --arg t "$TEMPLATE_ID" \
    '.[] | select(.id == $t)
    | .instance_configurations[]
    | {id: .id, type: .instance_type, node_types: .node_types, sizes: .discrete_sizes.sizes}'

Step 4 — Generic Terraform Template

Use the values from Steps 1–2 to fill in this generic template. Works for any hyperscaler and any deployment type — just swap the region, template ID, and instance config IDs.

HCL main.tf — generic template 
# ── Variables ──────────────────────────────────────────────────────────────
# Set these based on output from the API discovery commands above

locals {
  region      = "azure-australiaeast"  # ← Step 3 region code
  template_id = "azure-general-purpose" # ← Step 1 non-legacy template id

  # Instance config IDs from Step 2

  version_regex = "latest"
  region        = local.region
}

# ── Deployment ──────────────────────────────────────────────────────────────

  name                   = "my-deployment"
  region                 = local.region
  version                = data.ec_stack.latest.version
  deployment_template_id = local.template_id

  elasticsearch = {
    hot = {
      instance_configuration_id = local.ic_hot
      size                      = "8g"
      size_resource             = "memory"
      zone_count                = 2
      autoscaling               = {}
    }
  }

  kibana = {
    instance_configuration_id = local.ic_kibana
    topology = {
      size          = "1g"
      size_resource = "memory"
      zone_count    = 1

  }

  integrations_server = {
    instance_configuration_id = local.ic_fleet
    topology = {
      size          = "1g"
      size_resource = "memory"
      zone_count    = 1
    }
  }
}

One-Shot Discovery Script

Save this as discover.sh to interactively browse any region and template in one go:

BASH discover.sh 
#!/bin/bash
# Usage: EC_API_KEY=your-key ./discover.sh azure-australiaeast azure-general-purpose
# Arg 1: region code   Arg 2: template id (optional, lists all if omitted)

REGION=${1:?"Usage: $0  [template_id]"}
TEMPLATE=${2:-}
BASE_URL="https://api.elastic-cloud.com/api/v1/deployments/templates?region=$REGION"

if [ -z "$TEMPLATE" ]; then
  # List all non-legacy templates
  echo "=== Non-legacy templates for $REGION ==="
  curl -s -H "Authorization: ApiKey $EC_API_KEY" "$BASE_URL" \
    | jq '[.[] | select(.metadata != null)
      | select((.metadata | map(select(.key=="legacy" and .value=="true"))) | length == 0)
      | {id: .id, name: .name}]'
else
  # Show instance configs for specific template
  echo "=== Instance configs for $TEMPLATE in $REGION ==="
  curl -s -H "Authorization: ApiKey $EC_API_KEY" "$BASE_URL" \
    | jq --arg t "$TEMPLATE" \
      '.[] | select(.id == $t)
      | .instance_configurations[]
      | {id: .id, type: .instance_type, node_types: .node_types, sizes: .discrete_sizes.sizes}'
fi
BASH usage examples 
chmod +x discover.sh

# List all valid templates for Azure Australia East
EC_API_KEY=$EC_API_KEY ./discover.sh azure-australiaeast

# Get instance configs for a specific template
EC_API_KEY=$EC_API_KEY ./discover.sh azure-australiaeast azure-general-purpose

# AWS Sydney
EC_API_KEY=$EC_API_KEY ./discover.sh ap-southeast-2

# GCP Frankfurt
EC_API_KEY=$EC_API_KEY ./discover.sh gcp-europe-west3
ℹ️
Make This Part of Your CI Pipeline

Run the discovery commands as a pre-flight check before terraform plan in your CI pipeline. If a template ID disappears from the non-legacy list, your pipeline will catch it before Terraform attempts a failing apply.
